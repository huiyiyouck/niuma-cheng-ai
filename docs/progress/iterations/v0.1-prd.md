## 1. 迭代目标

把 `news-l1` 从 stub 做成**真实可用的 L1 处理**，交付 REQ-001 真实化。实现路径：**先打通最小端到端真实骨架（单条证据真实跑通），再逐项把五项能力填满 xiaobao 的真实需求**——先稳后全，而非一次堆齐。架构遵循 REQ-002 调研结论（确定性条件图编排、LLM 客户端内核移植、不做强可重入）。

## 2. 用户故事

- **US-1**（评分）作为调用方 xiaobao，我想投入单条新闻证据、取回真实的四维评分与理由，以便据此计算加权 `score_total`。
- **US-2**（标签）作为 xiaobao，我想取回五类标签，以便分类与检索。
- **US-3**（摘要）作为 xiaobao，我想取回真实摘要，以便列表/卡片展示。
- **US-4**（翻译）作为 xiaobao，我想对外文新闻取回目标语言译文，以便中文呈现。
- **US-5**（按需工具）作为 xiaobao，当证据不足且未预取上下文时，我想让 ai 在限额内按需补充上下文（库内检索/链接读取/Web 搜索），以便评分与摘要更准。
- **US-6**（可靠性）作为 ai Owner，我想在某家 LLM 故障时服务自动切到备用，以便整批处理不因单点失败。
- **US-7**（生态预留）作为未来的第二个内部调用方，我想通过统一的任务类型入口接入同一处理中枢——**v0.1 仅预留接口形状，不实现第二任务**。

## 3. 验收标准

> 每条覆盖正常 + 至少一个异常路径，可验证。PM 不自审本节，由 Architect / DevOps 审。

- **AC-1 评分**：当投入合法新闻证据时，应返回四个维度（时效、影响力、可信度、清晰度）各带 0–5 分值与非空理由；当内容为空或无法评分时，应按降级策略返回（默认值 + 说明），不返回 5xx。
- **AC-2 标签**：当处理成功时，应返回五类标签结构，且处理类标签体现**真实引擎标识**（不再是 stub 占位）。
- **AC-3 摘要**：当处理成功时，应返回非空摘要，符合约定的语言与长度。
- **AC-4 翻译**：当原文非目标语言时，应返回目标语言译文；当原文已是目标语言时，按约定可空或回填，不报错。
- **AC-5 按需工具**（v0.1 范围 = ai 能独立完成的部分）：
  - 当调用方已预取上下文（`kb_results`/`link_content`/`search_summary`）时，应正确**消费**作为输出依据；
  - 当未预取且证据不足时，应在工具调用上限内按需触发 ai 自给的工具：**链接读取**从 `raw_content` 约定 key（`url` / `canonical_url`，不新增 typed 字段、保持契约不变）取 URL 后 HTTP 自抓；**Web 搜索**用 **Tavily**（Owner 提供 key）；
  - **`tool_summary` 只统计 ai 主动工具调用** —— 调用方预取命中的 `kb_results`/`link_content`/`search_summary` **不计入** `tool_summary`（v0.1 在 xiaobao KB 实时能力落地前，`kb_search=0` 表示 ai 未主动调用 KB）；
  - 引用只保留**搜索结果中真实出现**的来源、不编造链接；
  - ai 无法自给的**主动库内检索（KB search）依赖 xiaobao 能力**（见前置依赖与 O-3），在其落地前以 `needs_context` 标记缺口、不阻塞其余处理。
- **AC-6 优雅降级**（对齐 v1 契约，不改契约）：当某步骤 LLM 失败但仍有部分可用结果时，应以 **`status=succeeded`** 返回，并在 `needs_context` / `tags.processing` / 对应维度理由中标注降级；**仅当完全不可用时**返回 `status=failed` + `output=null` + `error`。不在 `failed` 响应里塞 `output`，不一律抛 5xx。
- **AC-7 多 provider fallback**：当主 LLM 不可用（限流/超时/错误/空响应）时，应自动切换到下一个已配置 provider 重试；全部失败才返回失败（失败响应对齐 AC-6 语义）。provider 列表配置格式、重试顺序、失败判定矩阵（哪些情形计入失败、空响应是否计入）、每 provider timeout 继承等**最小配置形状由设计阶段 Architect 落定**（见 O-5），以便 Developer 按 TDD 覆盖限流/超时/错误/空响应。
- **AC-8 契约一致**：响应应符合 `news-l1` v1 契约（输入/输出字段与语义）；本迭代不改契约，如需改另走契约流程。
- **AC-9 生态薄接缝（内部 registry，对外契约不变）**：应提供**代码层**按任务类型分发的内部 registry / dispatch 结构（task_type → graph），`news-l1` 为首个注册项；**对外 HTTP 契约保持 `POST /v1/runs/news-l1` 不变、不新增 `caller`/`task_type` 传输字段**；v0.1 不实现第二个任务类型，仅验证内部 registry 存在且可扩展。未来要暴露 `/v1/runs/{task_type}` 路由或 caller 透传字段时，**另走契约变更流程**。

## 4. 范围边界

**本迭代做：**
- news-l1 五项能力**全量**真实化：评分 / 标签 / 摘要 / 翻译 / 按需工具调用。
- 服务健壮性：多 provider 链式 fallback、优雅降级、引用校验。
- 生态薄接缝：**代码层内部** registry / dispatch（task_type → graph），对外契约保持 `/v1/runs/news-l1` 不变——**只定义内部形状，不实现第二任务、不动对外契约**（caller 透传若需要另走契约变更）。
- 实现路径：先最小端到端真实骨架 → 逐项填满。

**本迭代不做：**
- L0 分类（归属 xiaobao）。
- 评分加权 `score_total`（留 xiaobao）。
- 强可重入 checkpoint。
- RunRecord 持久化审计记录（**延后 v0.2**）。
- 第二个任务类型 / 第二个调用方的具体实现（仅留接口形状）。
- 契约变更。
