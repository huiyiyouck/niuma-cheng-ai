## 1. 概述

v0.1 将 `news-l1` 从占位流水线改为真实 L1 处理。对外仍保持 coordination `news-l1` v1 契约不变：

- Endpoint 保持 `POST /v1/runs/news-l1`
- `L1Input` / `L1Output` / `RunResponse` 字段不增删改
- `failed` 响应保持 `output=null`
- `tasks` 仍是 xiaobao 侧业务真源，ai 只处理单条证据

内部实现采用确定性条件图，不引入自主 ReAct 循环。核心结构：

```text
FastAPI endpoint
  -> task registry dispatch("news-l1")
  -> news-l1 conditional graph
      -> ingest_context
      -> maybe_link_read
      -> maybe_web_search
      -> llm_process
      -> normalize_output
  -> RunResponse
```

v0.1 不实现第二个任务类型，但会引入内部 registry，使未来新增 task 时注册新图而不是改入口框架。

## 2. 数据模型

### L1Input / L1Output / RunResponse

来源：coordination `contracts/news-l1.md` v1 与 `src/agent_hub/schemas.py`。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `L1Input.raw_content` | `dict` | 保持 v1 契约不变 | 链接自抓 URL 从 `url` 或 `canonical_url` 读取 |
| `L1Input.kb_results` | `list[dict]` | 预取上下文 | 只作为输入依据，不计入主动 `tool_summary.kb_search` |
| `RunResponse.status` | `"succeeded" \| "failed"` | 不新增状态 | 部分可用结果为 `succeeded`；完全不可用为 `failed` |
| `RunResponse.output` | `L1Output \| None` | failed 时为 `None` | 不在 failed 中返回部分 output |
| `ToolSummary` | 3 个 int 字段 | 只统计 ai 主动工具调用 | 预取命中不计数 |

### TaskSpec

内部模型，不暴露到 HTTP。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `task_type` | `str` | v0.1 固定 `news-l1` | 内部 registry key |
| `input_model` | `type[BaseModel]` | v0.1 为 `L1Input` | 用于未来扩展，不改当前 endpoint |
| `graph` | `CompiledGraph` | 必填 | 对应 LangGraph 实例 |
| `init_state` | `Callable` | 必填 | 从 request 构造内部状态 |

### L1State

LangGraph 内部状态。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `run_id` | `str` | 必填 | endpoint 生成并贯穿日志 |
| `inp` | `L1Input` | 必填 | 原始请求 |
| `context_items` | `list[ContextItem]` | 默认空 | 预取、link、search 的归一化上下文 |
| `tool_summary` | `ToolSummary` | 默认 0 | 只统计 ai 主动调用 |
| `tool_budget_used` | `int` | `<= max_tool_calls` | 主动 link/web/KB 调用计数 |
| `needs_context` | `bool` | 默认 false | 输出时写入 `L1Output.needs_context` |
| `degradations` | `list[str]` | 默认空 | 写入 `tags.processing` 和理由 |
| `errors` | `list[StepError]` | 默认空 | 用于失败响应和降级说明 |
| `llm_result` | `LLMResult \| None` | 默认空 | LLM 原始结构化输出 |
| `output` | `L1Output \| None` | 默认空 | 规范化后的响应体 |

### ContextItem

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `source_type` | `"kb" \| "link" \| "web" \| "raw"` | 必填 | 上下文来源 |
| `title` | `str \| None` | 可空 | 搜索结果或原文标题 |
| `url` | `str \| None` | 可空 | link/web 来源 URL |
| `content` | `str` | 非空后才进入 prompt | 正文或摘要 |
| `metadata` | `dict` | 默认空 | provider、rank、raw 等 |

### ProviderConfig

从环境变量解析，不在 API 中暴露。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `name` | `str` | 唯一 | provider 标识，写入降级日志 |
| `base_url` | `str` | OpenAI 兼容 `/v1` base URL | 不含密钥 |
| `api_key_env` | `str` | 指向环境变量名 | 不把 key 放进 JSON 配置 |
| `model` | `str` | 必填 | 调用模型 |
| `timeout_ms` | `int \| None` | 可继承 `LLM_TIMEOUT_MS` | 单 provider 超时 |
| `supports_response_format` | `bool \| None` | 默认未知 | 首次 400 后可降级不发 |
| `temperature` | `float \| None` | 默认 0.2 | 不支持 temperature 时降级不发 |

环境变量建议：

```bash
LLM_PROVIDERS_JSON='[{"name":"primary","base_url":"https://api.example.com/v1","api_key_env":"OPENAI_API_KEY","model":"model-a"},{"name":"backup","base_url":"https://api.backup.com/v1","api_key_env":"BACKUP_OPENAI_API_KEY","model":"model-b"}]'
LLM_TIMEOUT_MS=60000
TAVILY_API_KEY=
WEB_SEARCH_MAX_RESULTS=5
```

兼容策略：`LLM_PROVIDERS_JSON` 缺失时，继续读取现有 `OPENAI_BASE_URL` / `OPENAI_API_KEY` / `L1_LLM_MODEL` 作为单 provider，便于本地冒烟。

### StepError

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `step` | `str` | 必填 | `link_read` / `web_search` / `llm_process` 等 |
| `kind` | `str` | 必填 | `timeout` / `provider_error` / `parse_error` 等 |
| `message` | `str` | 脱敏 | 不写密钥、完整 prompt |
| `recoverable` | `bool` | 必填 | 是否允许继续降级 |

## 3. API / 接口契约

### 对外 HTTP 契约

| 方法 | 路径 | 入参（类型） | 出参（类型） | 错误码 |
|------|------|--------------|--------------|--------|
| `GET` | `/health` | 无 | `{"status":"ok","service":"niuma-cheng-ai"}` | 5xx 仅服务异常 |
| `POST` | `/v1/runs/news-l1` | `L1Input` | `RunResponse` | 200 + `status=failed` 表示处理失败；请求体验证错误仍为 FastAPI 422 |

不新增 `/v1/runs/{task_type}`，不新增 caller/header 字段。内部 registry 不改变外部契约。

### 内部 task registry

```python
def get_task(task_type: str) -> TaskSpec: ...
def run_task(task_type: str, run_id: str, inp: BaseModel) -> TaskRunResult: ...
```

v0.1 只注册：

| task_type | input | output | graph |
|-----------|-------|--------|-------|
| `news-l1` | `L1Input` | `L1Output` | `news_l1_graph` |

### LLM client

```python
class AIClient(Protocol):
    def complete_json(self, messages: list[dict], timeout_ms: int) -> LLMResult: ...

class ChainedAIClient:
    def complete_json(self, messages: list[dict], timeout_ms: int) -> LLMResult: ...
```

Fallback 判定矩阵：

| 情形 | 当前 provider 是否结束 | 是否尝试下一个 provider | 记录 |
|------|------------------------|--------------------------|------|
| HTTP 429 / quota / rate limit | 是 | 是 | `provider_rate_limited` |
| HTTP 408 / timeout / 网络错误 | 是 | 是 | `provider_timeout` |
| HTTP 5xx | 是 | 是 | `provider_5xx` |
| HTTP 401 / 403 | 是 | 是 | `provider_auth_or_forbidden` |
| 空响应 | 是 | 是 | `provider_empty_response` |
| JSON 解析失败 | 先本地修复一次，仍失败则结束 | 是 | `provider_parse_error` |
| 不支持 `response_format` / `temperature` | 调整参数重试同 provider 一次 | 否，除非重试后仍失败 | `provider_quirk_adjusted` |
| 所有 provider 失败 | 全部结束 | 否 | `llm_all_providers_failed` |

约束：

- 每个 provider 默认一次业务尝试；provider quirk 的参数调整允许同 provider 追加一次。
- 总耗时不得超过 `min(inp.options.timeout_ms, service_timeout_budget)`。
- provider 错误信息必须脱敏，不记录 API key 或完整 prompt。

### Tool adapters

```python
def extract_url(raw_content: dict) -> str | None: ...
def read_url(url: str, timeout_ms: int) -> ToolResult: ...
def search_web(query: str, max_results: int, timeout_ms: int) -> ToolResult: ...
```

| 工具 | v0.1 状态 | 计数规则 | 失败处理 |
|------|-----------|----------|----------|
| KB prefetch | 已支持消费 | 不计入 `kb_search` | 空则继续 |
| KB realtime | adapter 接口占位，默认禁用 | 真正调用后才 `kb_search += 1` | 未配置时 `needs_context=true` |
| Link read | 从 `raw_content.url` / `canonical_url` 抓取 | 成功发起 HTTP 即 `link_read += 1` | 失败可降级继续 |
| Web search | Tavily adapter | 成功发起 search 即 `web_search += 1` | 失败可降级继续 |

## 4. 核心流程

### 4.1 请求入口

1. FastAPI 校验 `L1Input`。
2. 生成 `run_id`。
3. 调用 `run_task("news-l1", run_id, inp)`。
4. 成功且 `output` 非空：返回 `status=succeeded`。
5. 图返回完全不可用或抛出未恢复异常：返回 `status=failed, output=null, error=<脱敏摘要>`。

### 4.2 状态初始化

`init_news_l1_state()` 执行：

1. 把 `raw_text`、`raw_content.title`、预取的 `kb_results` / `link_content` / `search_summary` 归一化为 `context_items`。
2. 初始化 `tool_summary=0`，预取内容不计数。
3. 初始化 `tool_budget_used=0`。
4. 从 `raw_content.url` 或 `raw_content.canonical_url` 提取候选链接。

### 4.3 条件图路由

```text
START
  -> ingest_context
  -> decide_link_read
      -> link_read      (有 URL、无 link_content、预算足、上下文不足)
      -> decide_web_search
  -> decide_web_search
      -> web_search     (无 search_summary、预算足、上下文仍不足、Tavily 已配置)
      -> llm_process
  -> llm_process
  -> normalize_output
  -> END
```

上下文不足的 v0.1 启发式：

- `raw_text` 去空白后少于 300 字，且
- `kb_results`、`link_content`、`search_summary` 均为空或有效内容总长度少于 500 字。

该启发式只决定是否尝试 link/web，不影响调用方预取上下文的消费。主动 KB search 在 xiaobao 实时接口契约落地前不触发；缺 KB 时通过 `needs_context=true` 标记。

### 4.4 Link read

1. URL 只从 `raw_content.url` / `raw_content.canonical_url` 读取。
2. 仅允许 `http` / `https`。
3. 单次抓取超时建议 8 秒，最大正文输入 prompt 前裁剪到 12k 字符。
4. 抓取成功：加入 `ContextItem(source_type="link")`，`tool_summary.link_read += 1`。
5. 抓取失败：记录 recoverable error，继续 web search 或 LLM。

### 4.5 Web search

1. query 由 `raw_text` 标题片段、`raw_content.title`、`source_identity` 组合，最大 180 字符。
2. Tavily key 缺失时不调用，记录 `web_search_not_configured`，不计数。
3. 搜索结果归一化为 `ContextItem(source_type="web", title, url, content)`。
4. `L1Output.context` 只允许引用 tool adapter 返回的 URL，不接受 LLM 自造 URL。

### 4.6 LLM process

1. 构造单次结构化 JSON 任务：评分、标签、摘要、翻译、分析一次返回。
2. Prompt 明确：
   - 只输出 JSON；
   - 四维评分为 0-5 整数；
   - 每维 reason 非空；
   - 不编造来源 URL；
   - 原文已是中文时 `translation.zh` 可为空字符串。
3. `ChainedAIClient` 按 provider 顺序调用。
4. 解析失败时执行一次 JSON 修复；仍失败则尝试下一个 provider。
5. 全 provider 失败：若没有可构造的真实输出，返回完全失败；若只是工具失败但 LLM 成功，返回 succeeded 并标注降级。

### 4.7 Normalize output

输出清洗规则：

- `score` clamp 到 0-5。
- 缺失 reason 时填入降级说明。
- `title` 为空时用 `raw_content.title` 或 `raw_text[:40]`。
- `summary` 为空但 LLM 成功时按降级摘要填充并标注 `degraded:summary_empty`。
- `tags.processing` 至少包含：
  - `engine:agent_hub`
  - `llm:<provider_name>`
  - 有降级时追加 `degraded:<reason>`
- `needs_context=true` 的条件：
  - 主动 KB search 需要但未配置；
  - link/web 均不可用且上下文不足；
  - LLM 明确指出证据不足。

### 4.8 降级与失败语义

| 场景 | HTTP | RunResponse.status | output | error |
|------|------|--------------------|--------|-------|
| 全部成功 | 200 | `succeeded` | 完整 L1Output | null |
| link/web 失败但 LLM 成功 | 200 | `succeeded` | 带降级标记 | null |
| 主 provider 失败、backup 成功 | 200 | `succeeded` | 带 provider 标记 | null |
| 证据不足但能产出默认/保守结果 | 200 | `succeeded` | `needs_context=true` | null |
| 所有 LLM provider 失败且无可用输出 | 200 | `failed` | null | 脱敏失败摘要 |
| 请求 schema 非法 | 422 | FastAPI 默认 | 无 | FastAPI 默认 |

## 7. 风险和取舍

| 风险 / 取舍 | 决策 | 理由 | 后续动作 |
|-------------|------|------|----------|
| 自主 Agent vs 条件图 | 选确定性条件图 | 成本、延迟、测试可控 | 见 ADR-0001 |
| 自研 LLM client vs 移植 Horizon 内核 | 移植裁剪后的 OpenAI 兼容链式 client | 复用 provider quirk 和 fallback 经验 | 见 ADR-0002 |
| 通用 task endpoint | v0.1 不暴露 | 避免跨项目契约变更 | 内部 registry 先落 |
| failed 返回部分 output | 不允许 | 对齐 `news-l1` v1 | 部分可用统一 `succeeded` |
| KB realtime | 接口占位，默认禁用 | xiaobao 契约未落地 | 待 coordination 契约后接入 |
| Tavily API 细节 | adapter 隔离 | 外部 API 可能变化 | 实现前按官方文档核对 |

## 8. 测试要求

Developer 实现阶段至少覆盖：

1. `POST /v1/runs/news-l1` endpoint 仍兼容原契约。
2. 内部 registry 注册 `news-l1`，未知 task 抛内部错误但不暴露新 HTTP 路由。
3. 预取 `kb_results` / `link_content` / `search_summary` 被消费但不计入 `tool_summary`。
4. 有 URL 且上下文不足时触发 link read，并计入 `tool_summary.link_read`。
5. Tavily key 缺失时不触发 web search，返回可降级结果。
6. 主 provider 超时 / 429 / 5xx / 空响应时 fallback 到 backup。
7. 所有 provider 失败时返回 `status=failed, output=null`。
8. link/web 失败但 LLM 成功时返回 `status=succeeded` 且 `tags.processing` 含降级标记。
9. LLM 产生越界 score 时 normalize 到 0-5。
10. LLM 产出 context URL 必须来自工具结果，否则过滤。
