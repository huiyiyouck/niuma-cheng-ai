## Review 记录

### PRD R1 — Architect Review（2026-06-30）

- 结论：未通过（需 PM 修改后进入下一轮 Review）
- 核心问题：
  1. AC-9 / 范围边界涉及通用入口 `/v1/runs/{task_type}`、caller 标识，与 AC-8「本迭代不改契约」冲突。
  2. AC-6 “失败时返回部分可用结果”与 coordination `news-l1` v1 中 `failed => output=null` 的响应语义冲突。
  3. AC-5 链接自抓缺少 URL 来源约定；`tool_summary.kb_search` 需区分预取消费与 ai 主动工具调用。
- 建议：PM 修改 PRD，明确 AC-9 是内部薄接缝还是正式契约变更；将降级语义收敛到现有 v1 契约；补充 URL 来源和工具统计口径。

### PRD R1 — Developer Review（2026-06-30）

- 结论：未通过（需 PM 修改后进入下一轮 Review）
- 核心问题：
  1. AC-9 若新增通用路由或 caller 字段，会触发 API / schema / xiaobao 联调；若不改契约，应收敛为内部 registry。
  2. AC-6 未明确“部分可用结果”在 `RunResponse.status` 中算 `succeeded` 还是 `failed`，导致实现和测试无法稳定断言。
  3. AC-5 链接自抓缺少 URL 输入 key；`tool_summary` 需区分预取上下文消费与 ai 主动工具调用。
  4. AC-7 多 provider fallback 缺少最小配置形状和失败判定矩阵。
- 建议：PM 修改 PRD，给出可测试的状态语义、输入来源、统计口径和 AC-9 契约边界；provider 配置细节可要求 Architect 在设计阶段落定。

### PRD R2 — Developer Review（2026-06-30）

- 结论：通过
- 确认：
  1. AC-9 已收敛为内部 registry / dispatch，对外仍保持 `POST /v1/runs/news-l1`，不触发契约变更。
  2. AC-6 已明确部分可用与完全失败的 `RunResponse.status` 语义，可写稳定测试。
  3. AC-5 已明确 URL 来源 key 与 `tool_summary` 只统计 ai 主动工具调用。
  4. AC-7 已把 provider 配置形状、重试顺序、失败判定矩阵和 timeout 继承交给设计阶段落定，足够进入 PRD 定稿判断。
- 实现提示：现有骨架测试中预取 `kb_results` 计入 `tool_summary.kb_search` 的断言，需在实现阶段按 R2 PRD 新口径调整。

### PRD R2 — Architect Review（2026-06-30）

- 结论：通过，PRD 阶段定稿，可进入设计阶段。
- 确认：
  1. AC-9 已收敛为内部 registry / dispatch，对外契约仍为 `POST /v1/runs/news-l1`，不新增 caller/task_type 字段，避免跨项目契约变更。
  2. AC-6 已对齐 `RunResponse` v1：部分可用为 `succeeded`，完全不可用才 `failed + output=null + error`。
  3. AC-5 已明确 URL 来源 key 与 `tool_summary` 只统计 ai 主动工具调用，预取上下文不计入调用次数。
  4. AC-7 的 provider 配置结构、失败矩阵和 timeout 继承留给 Architect 设计阶段落定，作为 PRD 门禁已足够。
- 设计阶段注意：需落条件图编排与 LLM client 移植 ADR，并在设计文档中明确 provider fallback、`needs_context`、`tool_summary` 和内部 task registry 边界。
